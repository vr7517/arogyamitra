# ArogyaMitra

ArogyaMitra is a React-based medicine availability and pharmacy reservation prototype. It helps users search nearby pharmacies, compare medicine stock and prices, reserve medicines, and gives pharmacies/admins separate dashboards for operations and monitoring.

## Features

- Medicine search by medicine name, pharmacy name, or category.
- Filters for distance, availability, and maximum price.
- Sorting by distance, price, rating, or stock.
- Pharmacy detail page with available medicines, stock, price, call, WhatsApp, and map preview.
- User login and registration for patient and pharmacy accounts.
- Role-based protected dashboards for users, pharmacies, and admins.
- User dashboard for reservations, nearby pharmacies, alerts, and settings.
- Pharmacy dashboard for inventory management, medicine creation, stock updates, pricing, reservation status updates, low-stock alerts, history, analytics, and settings.
- Admin dashboard for platform analytics, pharmacy verification, user visibility, activity logs, and governance settings.
- Reservation flow with toast feedback and notification center.
- Dark mode support.
- Responsive UI built for desktop and mobile.
- Demo persistence using browser localStorage.

## Tech Stack

- React 19 for UI components.
- Vite 8 for local development and production builds.
- React Router 7 for page routing and protected routes.
- Tailwind CSS 4 through `@tailwindcss/vite` for styling.
- Lucide React for icons.
- Chart.js and React Chart.js 2 for admin analytics charts.
- React Context API for authentication, theme, reservations, notifications, and shared app state.
- ESLint for code quality checks.
- Browser localStorage for prototype data persistence.

## Frontend Management

The frontend is currently the main application layer.

- Routes are defined in `src/App.jsx`.
- Shared state is managed in `src/context/AppContext.jsx`.
- Seed data is stored in `src/data/dummyData.js`.
- Search results are generated by `src/hooks/useMedicineSearch.js` and `src/utils/liveData.js`.
- UI styling is handled with Tailwind utility classes and global styles in `src/index.css`.
- Protected pages use `src/components/ProtectedRoute.jsx`.

When adding a new frontend feature:

1. Add or update the page/component inside `src/pages` or `src/components`.
2. Add shared state actions to `AppContext.jsx` only if multiple pages need the same behavior.
3. Add route entries in `App.jsx` and route metadata in `src/routes/routes.js` when needed.
4. Keep seed/mock data in `dummyData.js` until the backend is connected.
5. Run `npm run lint` and `npm run build` before release.

## Database Management

This project does not currently use a real database. It uses:

- `src/data/dummyData.js` for default seed data.
- Browser localStorage for data created or changed while using the app.

Important localStorage keys:

- `arogya-registered-users` stores newly registered users and pharmacies.
- `arogya-user-id` stores the active logged-in user id.
- `arogya-theme` stores light/dark mode.
- `arogya-reservations` stores medicine reservations.
- `arogya-notifications` stores notification data.
- `arogya-pharmacy-inventory-{pharmacyId}` stores per-pharmacy inventory updates.

To reset demo data in the browser, clear site data from DevTools or run:

```js
localStorage.clear()
```

For a production database, move these entities into persistent tables or collections:

- `users`: name, email, password hash, role, locality.
- `pharmacies`: owner user id, name, address, contact, WhatsApp, coordinates, rating, verification status, opening hours.
- `medicines`: name, category, demand level, metadata.
- `inventory`: pharmacy id, medicine id, stock, price.
- `reservations`: user id, pharmacy id, medicine id, quantity, status, timestamps.
- `notifications`: user id, title, message, type, read status, timestamp.
- `activity_logs`: admin/audit events.

Recommended database choices:

- PostgreSQL for a relational production setup.
- MongoDB if you prefer document-based pharmacy/inventory records.
- Firebase/Supabase if you want faster hosted auth and database setup.

## Backend Management

There is no backend server in this repository yet. The file `src/utils/apiPlaceholders.js` documents the intended API boundaries:

```js
export const apiRoutes = {
  auth: '/api/auth',
  medicines: '/api/medicines',
  pharmacies: '/api/pharmacies',
  inventory: '/api/inventory',
  reservations: '/api/reservations',
}
```

A future backend should provide:

- Authentication: register, login, logout/session validation.
- User management: role-based access for user, pharmacy, and admin.
- Pharmacy management: create/update pharmacy profiles and verification.
- Medicine management: create/search medicines.
- Inventory management: update stock and prices by pharmacy.
- Reservation management: create reservations and update statuses.
- Notifications: send reservation and stock alerts.
- Analytics: expose dashboard totals, trends, and activity logs.

Suggested backend flow:

1. Create a backend folder such as `server/`.
2. Use Node.js with Express, NestJS, or another API framework.
3. Connect the backend to PostgreSQL, MongoDB, Firebase, or Supabase.
4. Replace direct `dummyData.js` and localStorage reads with API calls.
5. Implement the routes listed in `apiPlaceholders.js`.
6. Store secrets in environment variables, not in source code.
7. Add authorization middleware so only pharmacies can manage their inventory and only admins can verify pharmacies.

## Current Data Flow

1. `dummyData.js` provides default medicines, pharmacies, inventory, users, reservations, and analytics.
2. `AppContext.jsx` loads user/session/reservation/notification state from localStorage.
3. `liveData.js` merges seed data with localStorage updates.
4. Search pages and pharmacy pages read from `liveData.js`.
5. Pharmacy dashboard writes inventory changes back to localStorage.
6. Reservation actions update context state and then persist to localStorage.

## Notes

This is a frontend prototype, so passwords are stored as plain text in seed/local demo data. Do not use this auth model in production. A production backend must hash passwords, validate input, protect API routes, and store data in a secure database.
